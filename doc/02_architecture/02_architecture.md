【システム・アーキテクチャと設計思想】
保守費削減とユーザー環境での安定稼働を最優先とし、以下の技術方針を厳守すること。

1. 責務の分離（UIとロジックの完全分離）
- `internal/config` などのコアロジック層は、GUIライブラリ（walkやwinapi等）に直接依存させない。
- ロジック層のエラーや警告は呼び出し側に `error` として返し、ダイアログ表示の責務はUI層（`internal/ui/` 等）に完全に委譲すること。

3. 自己診断（Health Check）と自動復旧
- ユーザーの誤操作による設定ファイル破損や不意の強制終了に備え、起動時に設定のバリデーションや一時フォルダ（temp）の確実なクリーンアップを行う自己修復機能を組み込む。

4. リソースのハードリミット設定
- 低スペックPCでのフリーズを防ぐため、一定のメモリやディスク容量を消費した際に自動で処理制限やクリーンアップを行う安全装置をデフォルトで組み込む。

5. 命名規則・実装ルール
- ディレクトリ階層は最大3階層まで。
- 一つのGoファイルには1000行までしか記述しない。
- 既存の変数名・定数名の意図を汲み、無闇に変更しないこと。マジックナンバーやシステム制御のハードコードは禁止し、設定ファイルや `constants.go` で一元管理する。
